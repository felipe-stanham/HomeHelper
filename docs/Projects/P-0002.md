## Scope 3: Postgres Provisioning & Migrations

**Branch:** `scope/postgres-provisioning`
**Status:** `[DONE]`
**Depends on:** Scope 2

Implement the DB Provisioner module and integrate it into the app discovery flow. This is the heaviest infrastructure scope.

### Tasks
- [x] Add `postgres` section to platform config schema (host, port, superuser, superuser_password, database_prefix, role_prefix)
- [x] Create `pg_client.py` — sync Postgres client for superuser operations (create role, create database, grant/revoke privileges, run SQL)
- [x] Create `db_provisioner.py` module implementing the DB Provisioning workflow from `workflows.md`:
  - Generate secure random password for app role
  - CREATE ROLE with LOGIN
  - CREATE DATABASE with OWNER
  - REVOKE CONNECT FROM PUBLIC
  - GRANT CONNECT to app role
  - Create `schema_versions` table in app database
- [x] Implement migration runner:
  - List `migrations/` directory, sort numerically by prefix
  - Query `schema_versions` for already-applied migrations
  - Run pending migrations in a single transaction
  - Record each migration in `schema_versions` with checksum and timestamp
  - On failure: ROLLBACK, drop database and role (clean slate), return error
- [x] Integrate DB provisioner into App Manager discovery flow: after dependency check, before port allocation, call provisioner if `database: true`
- [x] Store `DatabaseInfo` (database_name, role_name, connection_url, applied_migrations) in registry
- [x] Update Service Manager app launch to pass `--db-url` when `database: true`
- [x] Update systemd service template to include `--db-url` parameter
- [x] Implement version bump migration flow: detect version change → diff migrations → stop app → run pending → restart
- [x] Handle migration failure on version bump: keep app stopped, log error with migration file and Postgres error
- [x] Add platform startup Postgres connectivity check (warn if unreachable, DB apps will fail)
- [x] Add unit tests for: DB provisioning, migration runner (success, failure, partial), version bump migration, schema_versions tracking
- [x] Add integration test: full lifecycle of an app with database (discover → provision → migrate → launch → version bump → re-migrate)

### Acceptance Criteria

- **cap-003**: When an app with `database: true` is first discovered, a new Postgres database named `latarnia_{app_name}` is created with a dedicated role `latarnia_{app_name}_role`. The role has CONNECT privilege only on its own database. The connection string is passed to the app as `--db-url` at launch. The database and role are recorded in the platform registry.
- **cap-004**: On first discovery of an app with `database: true` and a `migrations/` directory, all migration files are executed in order. A `schema_versions` table is created in the app's database tracking each applied migration. On version bump, only pending migrations run. If any migration fails, the app is NOT started and an error is logged with the failing migration file and Postgres error message. No partial state is left behind (migration runs in a transaction).

### Rabbit Holes — Do NOT Build
- Database backup or restore tooling
- Migration rollback automation — forward only
- Schema validation of migration SQL contents
- Connection pooling in the platform — apps manage their own connections

**Status:** `[IN PROGRESS]`

Implement Redis Streams setup and integrate into the discovery flow. This scope sets up the infrastructure; apps use streams directly at runtime.

### Tasks
- [x] Create `stream_manager.py` module implementing the Stream Setup workflow from `workflows.md`:
  - For each publish stream: check for publisher collision → create stream with MKSTREAM → record publisher in registry
  - For each subscribe stream: create stream if not exists → create consumer group per subscribing app → record in registry
- [x] Maintain a publisher ownership map in the registry (stream_name → publisher_app_id)
- [x] Integrate stream manager into App Manager discovery flow: after DB provisioning (if any), before dependency installation
- [x] Store `StreamInfo` (publish_streams, subscribe_streams, consumer_groups) in registry
- [x] Handle app unregistration: clean up consumer groups owned by the app (do NOT delete streams — other consumers may exist)
- [x] Update Redis channel naming convention: `latarnia:streams:{declared_name}`
- [x] Add unit tests for: stream creation, publisher collision detection, consumer group creation, app unregistration cleanup
- [x] Add integration test: two apps with publish/subscribe relationship — verify streams and consumer groups are created correctly

### Acceptance Criteria

- **cap-007**: When an app with `redis_streams_publish: ["crm.contacts.created"]` is registered, the stream `latarnia:streams:crm.contacts.created` exists in Redis. When a second app with the same publish stream is registered, registration fails with a collision error. Consumer groups are created per subscribing app. Apps can XADD, XREADGROUP, and XACK using their Redis connection.

### Rabbit Holes — Do NOT Build
- Schema registry for stream message contents
- Message validation or transformation in the platform
- Dead letter queue management — apps handle their own error cases
- Stream retention policies — use Redis defaults

**Status:** `[IN PROGRESS]`
**Depends on:** Scope 2

Define and document the contract for apps that expose MCP servers. Update the health check to include MCP liveness. This scope does NOT build the gateway — it builds the app-side contract and platform awareness.

### Tasks
- [x] Document the app-side MCP server contract in `System/app-specification.md`:
  - App runs an HTTP-based MCP server on its declared `mcp_port`
  - App must respond to MCP `list_tools` and `call_tool` requests
  - App must be ready to receive MCP requests after `/health` returns `good`
  - Recommended: use the `mcp` Python SDK with `sse` or `streamable-http` transport
- [x] Update Service Manager health check flow: after `/health` passes, if `mcp_server: true`, probe the MCP port with a basic HTTP request (or MCP `initialize` handshake)
- [x] Store MCP health status in registry `MCPInfo.healthy`
- [x] Update app launch sequence: log MCP port alongside REST port at startup
- [x] Create a minimal MCP server example (standalone Python file using `mcp` SDK) that can be included in the example app (Scope 9)
- [x] Add unit tests for: MCP health probe (success, failure, timeout)

### Acceptance Criteria

- **cap-005**: An app with `mcp_server: true` and `mcp_port: 9001` is expected to respond to MCP protocol requests on port 9001 after startup. The platform health check includes an MCP endpoint liveness probe after the standard `/health` check passes.

### Rabbit Holes — Do NOT Build
- The platform does NOT implement MCP for the app
- No MCP tool validation at the platform level (tools are the app's responsibility)
- No stdio transport support — HTTP only

## Scope 6: MCP Gateway

**Branch:** `scope/mcp-gateway`
**Status:** `[DONE]`
**Depends on:** Scope 5

The core differentiator. Build the platform-level MCP gateway that aggregates all app tools and exposes them to external clients.

### Tasks
- [x] Add `mcp` section to platform config (enabled, transport, gateway_path, tool_sync_interval)
- [x] Create `mcp_gateway.py` module:
  - Implement MCP server protocol (using `mcp` Python SDK) as a FastAPI route at `/mcp`
  - Maintain a tool index: map of `app_name.tool_name` → `{app_id, mcp_port, original_tool_name}`
  - On `list_tools`: return all tools from healthy MCP-enabled apps, namespaced as `app_name.tool_name`
  - On `call_tool`: parse namespace, lookup app, check health, proxy the call to the app's MCP server, return result
  - Return error if target app is unhealthy or unknown
- [x] Implement tool index rebuild:
  - On platform startup: query all healthy MCP-enabled apps, call `list_tools` on each, build index
  - On app start: fetch tools from the new app, add to index
  - On app stop: remove app's tools from index
  - On app version bump: fetch new tool list, verify backward compatibility (all prior tool names still present), update index. Reject version bump if tools were removed.
- [x] Integrate MCP gateway into FastAPI app startup (conditional on config `mcp.enabled`)
- [x] Store registered tool names in registry `MCPInfo.registered_tools`
- [x] Add unit tests for: tool index building, namespace parsing, health-aware routing, tool call proxying, backward compatibility check on version bump
- [x] Add integration test: MCP client connects to gateway, lists tools from multiple apps, calls a tool, receives result
- [x] **Feasibility validation**: Before full implementation, prototype the MCP server in FastAPI + proxy to a mock app MCP server. Validate SSE or Streamable HTTP transport works end-to-end.

### Acceptance Criteria

- **cap-006**: An MCP client connecting to `http://latarnia-host:8000/mcp` receives a tool listing containing all tools from all healthy MCP-enabled apps, namespaced as `app_name.tool_name`. A tool call to `crm.add_contact` is proxied to the CRM app's MCP server. If the CRM app is unhealthy, the tool call returns an error rather than hanging.
- **cap-011**: On version bump, if the previous version exposed MCP tools `[search, add, delete]` and the new version only exposes `[search, add]`, the version bump is rejected with a backward-compatibility violation error. Adding new tools `[search, add, delete, export]` is accepted.

### Rabbit Holes — Do NOT Build
- MCP authentication (OAuth, API keys) — v1 assumes trusted network
- Tool-level authorization (which client can call which tool)
- Hot-reload of tool surfaces — rebuild on app lifecycle events only
- Resource or prompt capabilities of MCP — tools only for v1

## Scope 7: Web UI Reverse Proxy

**Branch:** `scope/web-ui-proxy`
**Status:** `[DONE]`
**Depends on:** Scope 2

Implement the reverse proxy that lets apps serve their own web UIs through the platform.

### Tasks
- [x] Create `web_proxy.py` module:
  - FastAPI route matching `/apps/{app_name}/{path:path}`
  - Lookup app in registry: verify exists, `has_web_ui: true`, status is running
  - Proxy HTTP request to `http://localhost:{app_port}/{path}` using `httpx.AsyncClient`
  - Forward request headers, add `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Host`
  - Return response with original status code, headers, and body
- [x] Handle WebSocket upgrade: detect `Upgrade: websocket` header, proxy WebSocket connection
- [x] Handle error cases: app not found (404), app not running (503 with friendly HTML page), app has no web UI (404)
- [x] Path rewriting: strip `/apps/{app_name}` prefix before forwarding to app
- [x] Register proxy router in FastAPI app
- [x] Add unit tests for: path stripping, header forwarding, error responses
- [x] Add integration test: proxy to a simple HTTP server, verify HTML/CSS/JS/WebSocket pass through
- [x] **Feasibility validation**: Test `httpx`-based proxying with WebSocket upgrade and streaming responses before full implementation

### Acceptance Criteria

- **cap-008**: A request to `http://latarnia-host:8000/apps/crm/` is proxied to `http://localhost:{crm_port}/`. Static assets, WebSocket connections, and sub-paths are correctly proxied. The proxy adds appropriate headers (X-Forwarded-For, X-Forwarded-Proto). If the app is not running, the proxy returns a user-friendly error page.

### Rabbit Holes — Do NOT Build
- Caching layer for proxied content
- Authentication/session management for proxied apps
- HTTPS termination at the proxy level (app-level concern)
- Load balancing (single instance per app)
