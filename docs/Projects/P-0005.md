## Scope 1: ServiceManager hardening

**Branch:** `scope-P-0005-1-service-manager-hardening`
**Status:** `[DONE]`
**Depends on:** none

Fix the two bugs in `ServiceManager` that make its generated units unusable today, plus add the restart policy and linger warning. No routing change yet — the subprocess launcher still runs the show. This scope is testable in isolation by inspecting generated templates and unit tests; Scope 2 will wire the call sites.

### Tasks

- [x] Change `ServiceManager.generate_service_template` to use `sys.executable` (absolute path) for `ExecStart` instead of bare `python`. Cache the value on `ServiceManager` at `__init__` time.
- [x] Add `Environment=ENV={env}` to the generated unit's `[Service]` section. Use the same `env` value already computed for `service_prefix` (from P-0004).
- [x] Add `Restart=on-failure` and `RestartSec=5` to generated units, overridable by `manifest.config.restart_policy` when set (existing field, currently passed through verbatim).
- [x] Add a linger-check helper on `ServiceManager`: shell out to `loginctl show-user "$(whoami)" --property=Linger` on Linux. Expose as `ServiceManager.linger_enabled() -> bool`.
- [x] Call `linger_enabled()` at `main.py` startup on Linux. If false, emit a `WARNING` log: `"systemd --user linger is disabled for {user}. Per-app units may not survive logout. Enable with: sudo loginctl enable-linger {user}"`. Do not block startup.
- [x] Update `.claude/skills/deployment-process/SKILL.md` First-Time Bootstrap section to include `sudo loginctl enable-linger felipe` as a required step (before section "4. Enable and start").
- [x] Update `tests/unit/test_service_manager.py`:
  - Assert `ExecStart` contains the absolute path from `sys.executable`, not bare `python`.
  - Assert `Environment=ENV=dev` / `ENV=tst` / `ENV=prd` in generated template for each env.
  - Assert `Restart=on-failure` in default-generated template; assert override works when manifest sets `restart_policy`.
  - Mock `subprocess.run` to simulate linger on / off; assert the warning fires on "off".

### Acceptance Criteria

- **cap-001:** Generated template's `ExecStart` starts with an absolute Python path (`sys.executable`) followed by the app's `main_file`. Unit test verifies on both macOS and Linux (via `sys.executable` patching).
- **cap-002:** Generated template contains `Environment=ENV={env}` matching `ServiceManager.env`. Unit test covers all four cases: dev / tst / prd / fallback.
- **cap-006:** Generated template contains `Restart=on-failure` and `RestartSec=5` by default. When `manifest.config.restart_policy == "always"`, template contains `Restart=always`. When `restart_policy == "never"`, template contains `Restart=no`. Unit test covers all three.
- **cap-007:** On Linux with linger disabled, platform startup emits a `WARNING` log naming the missing prerequisite and remediation command. Deployment skill First-Time Bootstrap section lists `sudo loginctl enable-linger felipe` as a required step. Unit test covers the `loginctl` parse + warning path.

## Scope 2: Route service apps through systemd on Linux

**Branch:** `scope-P-0005-2-systemd-routing`
**Status:** `[DONE]`
**Depends on:** Scope 1

Reroute auto-start, the dashboard lifecycle buttons, and the `/api/apps/{id}/process/*` endpoints to `ServiceManager` when running on Linux with a `service`-type app. Leave streamlit apps and macOS on their existing subprocess path. Rename `MacOSProcessManager` to `SubprocessLauncher`.

### Tasks

- [x] Add a thin routing helper in `main.py` (or a new `launch_router.py` module — call it a module not a class if it's stateless). Signature: `pick_launcher(app_entry) -> ServiceManager | SubprocessLauncher | StreamlitManager`. Rule: `(platform.system(), app.manifest.type)` → streamlit → `StreamlitManager`; Linux + service → `ServiceManager`; Darwin + service → `SubprocessLauncher`.
- [x] Rename `src/latarnia/managers/process_manager_macos.py` → `src/latarnia/managers/subprocess_launcher.py`. Rename class `MacOSProcessManager` → `SubprocessLauncher`. Update imports in `main.py`, tests, and `managers/__init__.py`.
- [x] In the boot `auto_start` loop, replace direct `macos_process_manager.start_app(...)` with `pick_launcher(app_entry).start_service(...)` (or whichever verb matches the returned launcher's API — verbs may need harmonizing). On Linux + service apps, this path lands in `ServiceManager`; on Darwin, in `SubprocessLauncher`.
- [x] Harmonize verbs: ensure `ServiceManager` and `SubprocessLauncher` both expose `start_service(app_id) / stop_service(app_id) / restart_service(app_id)` with the same return contract (`bool`).
- [x] Update `/api/apps/{id}/process/start|stop|restart` endpoints to route through `pick_launcher`. `/api/services/{id}/...` continues to always target `ServiceManager` (for ops use even on macOS, where it would error loudly — acceptable).
- [x] Verify on a fresh local boot (macOS): routing logic + tests confirm `SubprocessLauncher` is picked on Darwin; `systemctl` not invoked on the subprocess path. Live-boot of `example_full_app` deferred because it requires Postgres which is not installed on this dev host (spec allows fallback to routing-logic verification).
- [ ] Verify on the Pi (TST): after a `latarnia-tst.service` restart with `auto_start: true` enabled on `example_full_app`, `systemctl --user list-units 'latarnia-tst-*'` shows the app unit in `active` state, dashboard shows the app as running, `/api/apps` confirms healthy. *(Deferred to TST post-merge — spec acknowledges this in the regression section.)*
- [x] Update unit tests: `tests/unit/test_subprocess_launcher.py` covers the renamed class; `tests/unit/test_service_manager.py` gets a test that exercises `start_service` end-to-end against a mocked `subprocess.run`.
- [x] Update `docs/System/architecture.md` and `docs/System/workflows.md` to reflect the new routing (see P-0005 `architecture.md` for the target state). *(Delegated to doc-updater after scope completion.)*

### Acceptance Criteria

- **cap-003:** With `auto_start: true` on `example_full_app` and platform running on Linux, platform boot produces a `latarnia-{env}-example_full_app.service` unit in `active` state. `systemctl --user is-active` returns `active`. Dashboard and `/api/apps` report the app as running and healthy. No subprocess child of the platform exists for the service app (`systemd-cgls` shows the app under its own unit, not under `latarnia-{env}.service`).
- **cap-004:** On macOS with `auto_start: true` on `example_full_app`, platform boot spawns the app via `SubprocessLauncher.start_service`. `systemctl` is not invoked. Integration test on macOS: boot platform → `curl :8100/` returns HTTP 200 within 10s. Streamlit launch behavior (if exercised on either OS) is unaffected.

## Scope 3: Combined health in dashboard

**Branch:** `scope-P-0005-3-combined-health`
**Status:** `[DONE]`
**Depends on:** Scope 2

Make the dashboard status reflect both systemd state and app-reported `/health`, using a single batched `systemctl show` call per refresh. Preserve all current dashboard visuals.

### Tasks

- [x] Extend `HealthMonitor` with `get_systemd_states() -> Dict[app_id, ActiveState]` on Linux. Implementation: one `systemctl --user show --property=Id,ActiveState,SubState --type=service 'latarnia-{env}-*.service'` call, parsed into a dict. Cache the result for the health check interval.
- [x] On macOS, `get_systemd_states` returns an empty dict (no systemd) and the dashboard relies on subprocess liveness + `/health` alone, matching today's behavior.
- [x] Change `HealthMonitor`'s per-app health computation to combine: `_combine(systemd_state, health_response)` per the table in `workflows.md#flow-03`. Return an enum `{green, yellow, red, grey}` plus a `detail` string. (Exposed via `get_overall_status(app_id)`.)
- [x] Update `/api/apps` response schema: each app entry gets `overall_status` (the combined value) plus `overall_status_detail`; existing fields retained. Dashboard JS reads `overall_status` for the card color; falls back to existing logic if missing (graceful for a staged rollout).
- [x] Update dashboard JS badge rendering to the combined enum. No HTML/CSS structural changes; only the value source. Screenshot test (manual, in the PR): before/after of an app in each of the four states.
- [x] Add unit tests for `_combine`: parametrized over the full state matrix. Add tests for the `systemctl show` parser (fixture output).
- [x] Update `TESTS.md` regression entries for dashboard: added four entries under "## Combined Health (P-0005 cap-005)" covering stopped/failed/error/green states and the `/api/apps` payload shape.

### Acceptance Criteria

- **cap-005:** Stopping `example_full_app` via `systemctl --user stop latarnia-{env}-example_full_app.service` causes the dashboard card to show a red status within one health check interval. Starting Redis down while the app runs causes the card to flip to red (app reports `/health` error) while `systemctl is-active` still says `active`. Restoring Redis returns the card to green within one interval. The four-state rule table from `workflows.md` is exhaustively tested via parametrized unit tests.
